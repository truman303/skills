# Module boundaries

Skill behaviour: do **not** turn module boundaries on at scaffold time. Wait
until there are ≥2 libs in `libs/web/`. With one lib, the rule has nothing
to enforce and just adds noise to lint output.

When it's time, the steps are:

1. Tag every project (in its `project.json#tags`).
2. Replace the empty `depConstraints` placeholder in the workspace's
   `eslint.config.mjs` with the actual rules.
3. Run `npx nx run-many -t lint` and confirm.

## Why the runtime boundary isn't enough

The `libs/web/` vs `libs/dotnet/` split is a runtime boundary — TypeScript
can't import C# and vice versa, so accidental cross-stack imports are
impossible. **But** ESLint can still enforce intentional structure on the
TypeScript side. That's where `scope:` and `type:` tags come in.

Because the runtime boundary already exists, `scope:` is **per product**,
not per stack. The shape we recommend:

| Project                                            | Tags                                |
| -------------------------------------------------- | ----------------------------------- |
| `apps/<any>`                                       | `scope:app`                         |
| `libs/web/shared/ui/shared-ui`                     | `scope:shared`, `type:ui`           |
| `libs/web/shared/data-access/api-client`           | `scope:shared`, `type:data-access`  |
| `libs/web/<product>/<module>/feature/<name>`       | `scope:<product>`, `type:feature`   |
| `libs/web/<product>/<module>/data-access/<name>`   | `scope:<product>`, `type:data-access` |
| `libs/web/<product>/<module>/ui/<name>`            | `scope:<product>`, `type:ui`        |
| `libs/web/<product>/<module>/util/<name>`          | `scope:<product>`, `type:util`      |
| `libs/dotnet/<Pascal.Name>`                        | `scope:dotnet`, `type:domain`       |

The .NET projects don't *need* tags (no ESLint runs on `.cs`), but adding
them keeps `nx graph` legible (color-coded by scope) and lets you write
the cross-stack rule below for symmetry.

## The rules to write

In the workspace's `eslint.config.mjs`, replace the placeholder:

```js
{
  '@nx/enforce-module-boundaries': [
    'error',
    {
      enforceBuildableLibDependency: true,
      allow: ['^.*/eslint(\\.base)?\\.config\\.[cm]?[jt]s$'],
      depConstraints: [{ sourceTag: '*', onlyDependOnLibsWithTags: ['*'] }]
    }
  ]
}
```

…with this richer set:

```js
depConstraints: [
  // 1. Apps can depend on anything.
  { sourceTag: 'scope:app', onlyDependOnLibsWithTags: ['*'] },

  // 2. Each product depends on itself and shared. No cross-product imports.
  { sourceTag: 'scope:shared',  onlyDependOnLibsWithTags: ['scope:shared'] },
  { sourceTag: 'scope:katydid', onlyDependOnLibsWithTags: ['scope:katydid', 'scope:shared'] },
  // …one entry per product.

  // 3. Type rules. feature can depend on everything; ui/util/data-access
  // are forbidden from depending on feature; ui and util are forbidden
  // from depending on data-access.
  { sourceTag: 'type:feature',     onlyDependOnLibsWithTags: ['type:feature', 'type:ui', 'type:data-access', 'type:util'] },
  { sourceTag: 'type:data-access', onlyDependOnLibsWithTags: ['type:data-access', 'type:util'] },
  { sourceTag: 'type:ui',          onlyDependOnLibsWithTags: ['type:ui', 'type:util'] },
  { sourceTag: 'type:util',        onlyDependOnLibsWithTags: ['type:util'] },

  // 4. Cross-stack rule (symmetry; the runtime boundary already prevents
  //    actual imports, but this catches accidental relative-path imports).
  { sourceTag: 'scope:web',    notDependOnLibsWithTags: ['scope:dotnet'] }
]
```

The fourth rule needs every web project to also carry `scope:web` (in
addition to its product scope). Add it in each `project.json#tags`. The
double-tag overhead is small and pays off the first time someone tries to
relative-path-import across the boundary.

## Verifying

```sh
npx nx run-many -t lint
```

Expect zero violations on the freshly scaffolded workspace. Add a violation
on purpose (have `apps/demo-app` import from a non-existent `scope:other`
lib) and confirm the rule fires.

## When to revisit

- New product folder under `libs/web/` → add a `scope:<product>` line in
  rule 2.
- New `type:` value (e.g. `type:e2e-util`) → add a `scope:<type>` line in
  rule 3.
- New stack folder (e.g. `libs/python/`) → mirror rule 4.

This is the only ESLint config tweak the skill touches. Everything else
that ships in `eslint.config.mjs` is generated by `nx add @nx/angular`.
