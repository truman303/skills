# Angular app

Two paths land here:

1. **Demo branch** — copy `assets/demo/apps/demo-app{,e2e}` verbatim, run the
   identifier rewrite, done. The proxy / `provideHttpClient()` / `serve`-target
   `proxyConfig` are all already wired up in the bundled files.
2. **New-app branch** — generate a vanilla Angular app and overlay the
   pieces the generator doesn't add (proxy, HttpClient, e2e config wired to
   `serve`).

This file documents path 2. Path 1 is fully covered by the asset copy + the
rewrite script in `scripts/rewrite_identifiers.sh`.

## Generators that match this workspace's defaults

The bundled `nx.json#generators` block locks in Angular generator defaults
matching the demo:

```jsonc
"generators": {
  "@nx/angular:application": {
    "e2eTestRunner": "playwright",
    "linter": "eslint",
    "style": "scss",
    "unitTestRunner": "vitest-angular"
  },
  "@nx/angular:library": {
    "linter": "eslint",
    "unitTestRunner": "vitest-analog"
  },
  "@nx/angular:component": { "style": "css" }
}
```

So in a workspace generated by this skill, the explicit-flag form below is
redundant — the defaults already match. Pass the flags anyway for clarity
and to insulate the skill from future changes to `nx.json`.

```sh
npx nx g @nx/angular:application apps/<app-name> \
  --name=<app-name> \
  --e2eTestRunner=playwright \
  --unitTestRunner=vitest-angular \
  --linter=eslint \
  --style=scss \
  --prefix=app \
  --routing \
  --standalone \
  --no-interactive
```

`--routing` adds an empty `app.routes.ts`; `--standalone` (the default in
Angular 21) yields an `app.config.ts` instead of a NgModule.

## Three overlays the generator does NOT do

After the generator runs you have a working Angular app and an e2e project,
but with no API to talk to. Apply these three small patches.

### 1. `provideHttpClient()` in `app.config.ts`

Without this, `HttpClient` injection throws at runtime. The generated file
looks like:

```ts
export const appConfig: ApplicationConfig = {
  providers: [
    provideBrowserGlobalErrorListeners(),
    provideRouter(appRoutes),
  ],
};
```

Add the import and the provider:

```ts
import { provideHttpClient } from '@angular/common/http';

export const appConfig: ApplicationConfig = {
  providers: [
    provideBrowserGlobalErrorListeners(),
    provideRouter(appRoutes),
    provideHttpClient(),
  ],
};
```

The bundled `assets/demo/apps/demo-app/src/app/app.config.ts` is exactly this
shape.

### 2. `proxy.conf.json` + the `serve`-target `proxyConfig` option

The Angular dev server proxies `/api/*` → `http://localhost:5039` so the SPA
can call the .NET API without CORS in dev.

Create `apps/<app-name>/proxy.conf.json` (mirror
`assets/demo/apps/demo-app/proxy.conf.json`):

```json
{
  "/api": {
    "target": "http://localhost:5039",
    "secure": false,
    "changeOrigin": true,
    "logLevel": "debug",
    "pathRewrite": { "^/api": "" }
  }
}
```

`pathRewrite` strips the `/api` prefix so the upstream `.NET` route is plain
`/weatherforecast` — keeps the C# attribute routing identical to a server-only
deployment.

Then patch `apps/<app-name>/project.json` so the `serve` target picks up the
proxy config. Add `proxyConfig` under `targets.serve.options`:

```jsonc
"serve": {
  "continuous": true,
  "executor": "@angular/build:dev-server",
  "options": {
    "proxyConfig": "apps/<app-name>/proxy.conf.json"
  },
  "configurations": { ... }
}
```

The full target shape is in `assets/demo/apps/demo-app/project.json` for
reference.

### 3. Wire up the e2e project's `serve` target

If `--e2eTestRunner=playwright` was used, you already have `apps/<app-name>-e2e/`
with a `playwright.config.ts` whose `webServer.command` is
`npx nx run <app-name>:serve` and `url` is `http://localhost:4200`. Verify
`reuseExistingServer: true` is set so local runs don't conflict with a
hand-started `nx serve`.

The bundled `apps/demo-app-e2e/playwright.config.ts` shows the recommended
shape (Chromium + Firefox + WebKit projects, trace on first retry, `cwd:
workspaceRoot`). The `cwd: workspaceRoot` line matters: Playwright resolves
`webServer.command` relative to its own config file by default, but
`npx nx ...` only works from the workspace root.

## Verifying

From the workspace root (and any time you change either app):

```sh
npx nx serve <app-name>             # serves on http://localhost:4200
# in another shell:
curl http://localhost:4200/api/weatherforecast | head
```

If the curl returns JSON, the proxy is wired correctly. If it returns the
Angular `index.html`, `proxyConfig` is missing or pointing at the wrong file.
